애플리케이션을 설계할 때 자주 마주하는 고민 중 하나는 입력값 검증을 어느 계층에서 어디까지 수행해야 하는지이다.
모든 계층에서 같은 검증을 반복하면 비효율적이고, 반대로 한 곳에서만 검증하면 안정성이 떨어질 수 있다.
결국 각 계층의 책임을 분리하면서도 중복을 최소화하는 설계가 중요하다.

테스트에서도 비슷한 고민이 있다. 외부 의존성을 어떻게 대체할지에 따라 테스트의 목적과 신뢰성이 달라지기 때문이다.
이때 자주 사용되는 도구가 `Mockito`이며, 특히 Mock, Stub, Spy의 차이를 정확히 이해하는 것이 중요하다.

이 글에서는 먼저 애플리케이션의 각 계층에서 입력값 검증 책임을 어떻게 나눌 것인지,
그리고 중복 검증을 피하면서 안정성을 확보하는 방법과 그 트레이드오프를 살펴본다.
이어서 Mockito의 Mock, Stub, Spy 개념과 각각을 어떤 상황에서 선택해야 하는지를 예시와 함께 정리한다.

---

## 1. 애플리케이션 각 계층의 입력값 검증 범위와 책임

계층형 아키텍처에서는 일반적으로 **프레젠테이션 계층(Controller), 애플리케이션/서비스 계층(Service),
도메인 계층(Domain/Entity), 영속성 계층(Repository/DB)** 이 각기 다른 책임을 가진다.
입력값 검증도 이 책임에 맞게 나누는 것이 중요하다.

### Controller 계층의 검증

`Controller` 계층은 외부로부터 들어오는 요청의 형식과 기본 제약 조건을 검증하는 역할을 맡는 것이 적절하다.
다음과 같은 항목들이 여기에 해당한다.

- 필수값 누락 여부
- 문자열 길이 제한
- 이메일 형식
- 숫자 범위
- null/blank 여부

`Spring`에서는 보통 @Valid, @NotBlank, @Email, @Size 같은 `Bean Validation`을 활용한다.

```java
public record UserCreateRequest(
        @NotBlank String username,
        @Email String email,
        @NotBlank String password
) {}
```
이 계층의 목적은 “잘못된 형태의 요청”을 가능한 한 빨리 걸러내는 것이다. 즉, **사용자 입력 형식 검증** 이 중심이다.

### Service 계층의 검증

`Service` 계층은 단순 형식 검증이 아니라 비즈니스 규칙 검증을 담당해야 한다. 
다음과 같은 검증이 여기에 속한다.

- username이 이미 존재하는지
- 이미 삭제된 계정은 수정할 수 없는지
- 현재 사용자에게 해당 리소스를 수정할 권한이 있는지
- 주문 상태상 취소가 가능한지

```java
if (userRepository.existsByUsername(username)) {
    throw new IllegalArgumentException("Username already exists");
}
```
이 계층의 검증은 “이 요청이 비즈니스적으로 유효한가?”에 초점을 둔다.

### Domain 계층의 검증

도메인 계층은 객체 자체가 항상 유효한 상태를 유지하도록 보장하는 검증을 담당하는 것이 바람직하다.
예를 들어 `User` 객체라면 “username은 비어 있을 수 없다”, “displayName은 null일 수 없다” 같은 규칙은
엔티티/도메인 모델에서 스스로 지킬 수 있어야 한다.

즉, 도메인 계층은 다음을 보장해야 한다.
- 무효한 상태의 객체가 생성되지 않도록 함
- 상태 변경 시 불변 조건(invariant)을 유지함

이 계층의 검증은 외부 요청과는 무관하게 **객체의 정합성 자체** 를 지키는 역할을 한다.

### Repository / Database 계층의 검증

`Repository`나 `DB`는 마지막 방어선이다.
특히 다음과 같은 제약은 DB에서 반드시 보장해야 한다.

- unique 제약
- not null 제약
- foreign key 제약

예를 들어 서비스 계층에서 중복 검사를 했더라도, 동시성 상황에서는 두 요청이 동시에 통과할 수 있다.
따라서 최종적으로는 `DB`의 `unique constraint`가 안정성을 보장해야 한다.

즉, 서비스 검증은 사용자 친화적 에러 처리, **DB 제약은 최종 무결성 보장** 으로 보는 것이 적절하다.

---

### 중복 검증을 피하면서도 안정성을 확보하는 방법

입력값 검증에서 가장 중요한 것은 같은 책임의 검증을 여러 계층에 반복하지 않는 것이다.
예를 들어 이메일 형식 검증을 Controller, Service, Domain 모두에서 반복하면 코드 중복만 늘고 유지보수성이 떨어진다.

따라서 다음과 같이 책임을 분리하는 것이 현실적이다.
- Controller: 형식/스펙 검증
- Service: 비즈니스 규칙 검증
- Domain: 객체 불변식 검증
- DB: 최종 무결성 보장

이 구조를 따르면 중복을 최소화하면서도 안정성을 확보할 수 있다.

다만 완전히 중복을 없앨 수는 없다.
예를 들어 `username` 중복 검사는 `Service`에서 미리 검사해 사용자에게 친절한 메시지를 줄 수 있지만,
최종적으로는 `DB`에서도 `unique` 제약이 필요하다. 이것은 “중복”이라기보다 **역할이 다른 이중 방어**에 가깝다.

### 관련 트레이드오프

검증 책임을 분리하면 구조는 깔끔해지지만, 계층 간 경계가 늘어나면서 설계가 복잡해질 수 있다.
반대로 모든 검증을 한 계층에 몰아넣으면 구현은 단순해지지만, 책임이 섞이고 유지보수성이 떨어진다.

또한 중복 검증을 줄이면 코드가 간결해지지만, 특정 계층이 실패했을 때 안전망이 부족해질 수 있다.
반대로 여러 계층에서 방어하면 안정성은 올라가지만 코드 중복과 비용이 증가한다.

결국 실무에서는 다음과 같은 균형이 중요하다.

- 형식 검증은 Controller에서 빠르게
- 비즈니스 검증은 Service에서 명확하게
- 최종 무결성은 DB에서 강하게

즉, 중복을 완전히 제거하는 것보다 **“책임이 겹치지 않도록 검증 목적을 분리하는 것”** 이 핵심이다.

## 2. Mockito의 Mock, Stub, Spy 개념과 선택 기준

`Mockito`는 테스트에서 의존성을 제어하기 위해 사용하는 대표적인 라이브러리다.
그중 Mock, Stub, Spy는 비슷해 보이지만 목적과 사용 방식이 다르다.

### Mock

`Mock`은 **행위 검증(behavior verification)** 을 중심으로 사용하는 테스트 대역이다.
즉, 어떤 메서드가 “호출되었는지”, “몇 번 호출되었는지”, “어떤 인자로 호출되었는지”를 검증하는 데 초점이 있다.

예를 들어 회원 가입 시 이메일 발송 서비스가 호출되는지 확인하고 싶다면 `Mock`이 적절하다.

```java
EmailSender emailSender = mock(EmailSender.class);

userService.register(request);

verify(emailSender).sendWelcomeMail(anyString());
```
여기서 핵심은 “정말 메일이 전송됐는가”가 아니라, **메일 발송 메서드를 호출했는가를 확인** 하는 것이다.

### Stub 

`Stub`은 상태 기반 테스트(state verification) 를 위해 사용하는 테스트 대역이다.
즉, 어떤 메서드가 호출되었는지를 검증하기보다,
**정해진 값을 반환하도록** 미리 설정해서 테스트를 안정적으로 만드는 데 사용한다.

예를 들어 외부 API 대신 고정된 사용자 정보를 반환하게 만들고 싶다면 `Stub`이 적절하다.

```java
when(userRepository.findById(1L)).thenReturn(Optional.of(user));
```
이 경우 테스트의 목적은 findById()가 몇 번 호출되었는지가 아니라,
서비스가 특정 사용자 데이터를 받았을 때 올바르게 동작하는지 확인하는 것이다.

실제로 Mockito에서는 when(...).thenReturn(...) 형태로 만든 `Mock` 객체를 `Stub`처럼 사용하는 경우가 많다.
즉, `Mockito` 문맥에서는 `Mock`과 `Stub`이 도구적으로 겹쳐 쓰이지만, 반환값 제어가 중심이면 `Stub`이다.

### Spy

`Spy`는 실제 객체를 감싸되, 일부 동작만 감시하거나 오버라이드할 때 사용한다.
즉, `Mock`처럼 완전히 가짜 객체를 만드는 것이 아니라, 
진짜 객체의 동작을 기본으로 유지하면서 일부 메서드만 조작할 수 있다.

```java
List<String> list = new ArrayList<>();
List<String> spyList = spy(list);

spyList.add("A");
verify(spyList).add("A");
```
`Spy`는 실제 로직이 실행되기 때문에, 기존 객체의 동작을 유지하면서 호출 여부를 검증할 수 있다.

또한 특정 메서드만 바꾸고 싶을 때도 사용한다.
```java
doReturn(100).when(spyCalculator).calculate();
```

## 어떤 상황에서 어떤 방식을 선택해야 할까?

### Mock을 선택하는 경우

- 외부 의존성이 호출되었는지 검증하고 싶을 때
- 이메일 발송, 로그 기록, 이벤트 발행처럼 행위 자체가 중요한 경우
- 테스트 대상이 의존성과 어떻게 상호작용하는지 확인하고 싶을 때

예:
  - emailSender.send() 호출 여부 검증
  - eventPublisher.publish() 호출 여부 검증

### Stub을 선택하는 경우

- 의존성의 반환값만 필요하고, 호출 여부는 중요하지 않을 때
- 테스트를 위해 외부 의존성을 고정된 값으로 제어하고 싶을 때
- 서비스 로직의 결과를 검증하는 것이 목적일 때

예:
  - Repository가 특정 엔티티를 반환하게 설정
  - 외부 API 클라이언트가 고정 응답을 반환하게 설정

### Spy를 선택하는 경우
 
- 실제 객체 동작을 유지하면서 일부만 검증/변경하고 싶을 때
- 레거시 코드처럼 완전한 `Mocking`이 어려운 경우
- 일부 메서드는 실제로 수행하고, 일부 메서드만 조작하고 싶을 때

예:
  - 실제 컬렉션 객체의 동작을 유지하면서 호출 여부 검증
  - 서비스 일부 메서드만 overriding 해서 테스트

다만 `Spy`는 실제 메서드가 실행되기 때문에, 부작용이 있는 객체에 사용하면 테스트가 불안정해질 수 있다.
따라서 `Spy`는 `Mock`이나 `Stub`보다 더 신중하게 사용해야 한다.

---

## 정리

`Mockito`의 Mock, Stub, Spy는 모두 테스트 대역이지만 목적이 다르다.
  - Mock: 호출 여부와 상호작용 검증
  - Stub: 반환값 제어
  - Spy: 실제 객체 기반 부분 제어

실무에서는 대부분 `Mockito Mock` 객체를 생성한 뒤,
일부는 Stub처럼 사용하고, 필요한 경우 verify()로 Mock처럼 검증한다. 
따라서 도구 사용법보다 더 중요한 것은 **테스트의 목적이 무엇인지 먼저 명확히 하는 것**이다.
